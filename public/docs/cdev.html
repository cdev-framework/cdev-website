class cdev.cdev_objects.cdev_module(path)¶

Bases: object

Class the represents a python module that contains a serverless function

A python module can represents a namespace, which means that functions in the same module can reference and depend on symbols from the module. Serverless functions will be a first level function within the module, so any symbol created at the global level of the module can effect the lambda. Thus we must parse the global symbols created in the module and compare that to the serverless functions to create a dependency tree.

class cdev.cdev_objects.serverless_function(name, source_code, module)¶

Bases: object

A representation of a serverless function

A serverless function is an event triggerable function. The function should take the event as some kind of input (either directly or from middleware).

When a function is marked as a serverless function with the decorator, we must consider that the function can interact and depend on elements in its module. Therefore, to capture the entire context of a serverless function we must include any dependency it has in the module. A dependency can be thought of as a referencing a symbol from the symbol table. Once all dependecies are calculated, the function can be written into a seperate intermediate file that will be what is uploaded to the cloud service managing the function.

This dynamic parsing out of function allows functions to be logically grouped like previous forms of development without dealing with previous problems of serverless function management (https://github.com/cdk-patterns/serverless/blob/master/the-lambda-trilogy/README.md)
